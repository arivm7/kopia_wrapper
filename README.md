# Kopia Backup Scripts

Обёртка для [kopia](https://kopia.io). 
Набор из трёх bash-скриптов для резервного копирования и восстановления системных каталогов и файлов через [kopia](https://kopia.io) — быстрый, шифрующий, дедуплицирующий инструмент резервного копирования с открытым исходным кодом.

## Состав проекта

| Файл | Назначение |
|---|---|
| `kopia_common.sh` | Общая библиотека: пути, настройки, служебные функции. Не запускается самостоятельно, только подключается через `source`. |
| `kopia_backup.sh` | Создаёт снапшоты всех настроенных путей и применяет политику хранения. |
| `kopia_restore_to.sh` | Восстанавливает последние снапшоты всех путей в отдельную папку с сохранением структуры; умеет также показывать информацию об архиве без восстановления (`--info`). |
| `install.sh` | Копирует все файлы проекта в выбранную папку, настраивает права и подключает автодополнение. |
| `kopia-completion.bash` | Автодополнение bash (только имена флагов) для `kopia_backup.sh` и `kopia_restore_to.sh`. |

Все файлы должны находиться в одном каталоге — `kopia_backup.sh`, `kopia_restore_to.sh` и `install.sh` подключают `kopia_common.sh` из своей директории через `source "${APP_DIR}/kopia_common.sh"`.

## 1. `kopia_common.sh` — общая библиотека

### 1.1. Определение путей

Все пути привязаны к домашнему каталогу пользователя, который **реально** запустил скрипт, а не к тому, во что превращается `$HOME` после `sudo`:

```bash
if [[ -n "${SUDO_USER:-}" ]]; then
    REAL_HOME="$(getent passwd "$SUDO_USER" | cut -d: -f6)"
else
    REAL_HOME="$HOME"
fi
```

`SUDO_USER` всегда содержит имя исходного пользователя (устанавливается самим `sudo`), поэтому `REAL_HOME` остаётся стабильным независимо от того, была ли эскалация прав.

На основе `REAL_HOME` строятся все рабочие пути:

| Переменная | Путь | Назначение |
|---|---|---|
| `CONFIG_PATH` | `$REAL_HOME/.config/kopia_backup/` | Каталог с пользовательскими настройками |
| `CONFIG_FILE` | `$CONFIG_PATH/kopia_common.conf` | **Единый** конфиг, общий для `kopia_backup.sh` и `kopia_restore_to.sh` (имя берётся от `kopia_common.sh`, а не от вызывающего скрипта) |
| `REPO_PATH` | `$REAL_HOME/Backups/kopia_backup/<hostname>` | Физическое хранилище данных репозитория kopia (подпапка с именем хоста — удобно, если один и тот же конфиг когда-нибудь окажется скопирован на другую машину) |
| `KOPIA_CONFIG` | `$CONFIG_PATH/repository.config` | Файл подключения kopia к репозиторию |
| `KOPIA_PASSWORD_FILE` | `$CONFIG_PATH/.kopia_pass` | Пароль репозитория в открытом виде (для неинтерактивного запуска) |
| `RESTORE_ROOT_DEFAULT` | `$REAL_HOME/Backups/kopia_restore/<метка_времени>` | Папка восстановления по умолчанию |

### 1.2. `SOURCE_PATHS` — что бэкапится

Массив путей (каталогов и/или отдельных файлов), которые попадают в резервную копию:

```bash
SOURCE_PATHS=(
    "/etc"
    "/var/www"
    "/var/spool/cron"
    "/home"
    "/root"
)
```

kopia одинаково умеет создавать снапшоты как каталогов, так и отдельных файлов — можно смешивать оба типа в списке.

### 1.3. Исключения (`.kopiaignore`)

Для исключения подпапок/файлов из бэкапа конкретного пути используется файл `.kopiaignore` внутри самого этого каталога (синтаксис как в `.gitignore`):

```
cache/
tmp/
*.log
node_modules/
!important.log
```

Автоматического применения исключений через код в скриптах нет — это делается вручную, кладя `.kopiaignore` в нужные каталоги.

### 1.4. Механизм пользовательского конфига

При первом запуске любого из скриптов (`kopia_backup.sh` или `kopia_restore_to.sh`) вызывается `read_config_file()`:

- Если `$CONFIG_FILE` уже существует — он подключается через `source`, и значения переменных (`REPO_PATH`, `SOURCE_PATHS` и т.д.) переопределяются из него.
- Если файла ещё нет — вызывается `save_config_file()`, которая:
  1. Извлекает через `awk` фрагмент `kopia_common.sh` между маркерами `[CONFIG START]` и `[CONFIG END]` (это как раз блок с `REPO_PATH`, `KOPIA_CONFIG`, `KOPIA_PASSWORD_FILE`, `RESTORE_ROOT_DEFAULT`, `SOURCE_PATHS` и комментарием про `.kopiaignore`).
  2. Сохраняет этот фрагмент как есть в `$CONFIG_FILE` (`~/.config/kopia_backup/kopia_common.conf` — **один файл на оба скрипта**, поэтому изменения путей/списка бэкапируемых каталогов достаточно внести один раз).
  3. Выводит сообщение с предложением отредактировать конфиг и запустить скрипт заново, после чего завершает работу (`exit 0`).

Это выполняется **до** эскалации прав (`require_root`), поэтому файл создаётся и остаётся собственностью обычного пользователя — редактировать его можно без `sudo`.

### 1.5. Проверка и установка зависимостей

```bash
REQUIRED_PACKAGES=(
    "kopia:"   # без apt-пакета — устанавливается вручную по инструкции с kopia.io
    "jq:jq"    # доступен через apt
)
```

Формат: `"команда:apt_пакет"`. `check_required_packages()`:
- для программ без apt-пакета (пустая часть после `:`) — при отсутствии сразу выводит ошибку и завершает скрипт (`exit 1`), автоустановка не предлагается;
- для программ с apt-пакетом — при отсутствии спрашивает разрешение на `apt-get install -y`; при отказе или ошибке установки — тоже `exit 1`.

### 1.6. Проверка файла пароля

`check_kopia_password_file()` проверяет, что `$KOPIA_PASSWORD_FILE`:
1. существует;
2. имеет права доступа `600`.

При несоответствии — понятная ошибка с инструкцией по исправлению и `exit 1`.

### 1.7. Самоповышение прав (`require_root`)

```bash
require_root() {
    if [[ "$(id -u)" -eq 0 ]]; then
        return 0
    fi
    exec sudo -- "$0" "$@"
    ...
}
```

Если скрипт запущен не от root — он перезапускает **сам себя** через `sudo` с теми же аргументами командной строки (`exec` полностью замещает текущий процесс, не порождая дочерний). Благодаря этому:
- вручную вызывать `sudo` не нужно — достаточно запустить скрипт как обычный пользователь;
- все операции после `require_root` (чтение `/etc`, создание снапшотов и т.д.) выполняются от root;
- справка (`--help`/`--usage`) и версия (`--version`) отрабатывают без запроса пароля sudo, так как вызываются до `require_root`.

### 1.8. Прочие функции

| Функция | Назначение |
|---|---|
| `log_info` / `log_error` | Обёртки над `logger` — пишут сообщения в системный журнал с префиксом `BACKUPER`. |
| `exit_with_msg "текст" код` | Единая точка вывода сообщения (с цветом в терминале) + запись в лог + `exit`. Код `0` — успех, `1` — ошибка, `2` — ошибка с подсказкой по `--usage`, другое — информационное сообщение. |
| `print_version` | Печатает имя скрипта, заголовок, версию, историю изменений и copyright (переменные `APP_VERSION`, `LAST_CHANGES`, `COPYRIGHT` в начале файла). |

## 2. `kopia_backup.sh` — резервное копирование

### Ключи командной строки

```
kopia_backup.sh [-h|--help] [-u|--usage] [-v|--version]
```

| Ключ | Действие |
|---|---|
| `-h`, `--help` | Полная справка |
| `-u`, `--usage` | Краткая строка использования |
| `-v`, `--version` | Версия скрипта |

Позиционных аргументов скрипт не принимает.

### Порядок выполнения

1. Разбор `-h`/`-u`/`-v` (если указаны — вывод и выход, без root).
2. `read_config_file` — чтение/создание пользовательского конфига (без root).
3. Вывод диагностической информации: путь к репозиторию (`REPO_PATH`), конфиг-файл обёртки (`CONFIG_FILE`), конфиг kopia (`KOPIA_CONFIG`).
4. `require_root "$@"` — самоповышение прав через `sudo`, если нужно.
5. `check_required_packages` — проверка/установка `kopia`, `jq`.
6. `check_kopia_password_file` — проверка файла пароля.
7. Подключение к существующему репозиторию (`kopia repository connect`) или создание нового (`kopia repository create`), если он ещё не инициализирован по `$REPO_PATH`.
8. Цикл по `SOURCE_PATHS`:
   - `kopia snapshot create "$DIR"` — создание снапшота;
   - `kopia policy set "$DIR" --keep-daily 30 --keep-weekly 12 --keep-monthly 12` — политика хранения (30 суточных / 12 недельных / 12 месячных снапшотов).
   - Пути, которые не существуют на диске, пропускаются с предупреждением, а не обрывают весь скрипт.
9. `kopia maintenance run` — лёгкое обслуживание репозитория (компактификация индексов и т.п.).
   Полное обслуживание (`--full`) в скрипт **намеренно не входит** — его рекомендуется запускать отдельным cron-заданием раз в неделю:
   ```bash
   kopia maintenance run --config-file "$KOPIA_CONFIG" --full
   ```
10. Если хотя бы один путь завершился с ошибкой — скрипт завершается с кодом `1` и списком проблемных путей; иначе — `exit 0`.

Все вызовы `kopia` явно используют `--config-file "$KOPIA_CONFIG"`, чтобы гарантированно работать с нашим репозиторием, а не с конфигом kopia по умолчанию.

## 3. `kopia_restore_to.sh` — восстановление

### Ключи и аргументы

```
kopia_restore_to.sh [-h|--help] [-u|--usage] [-v|--version] [-i|--info [ПУТЬ]] [ПУТЬ_ВОССТАНОВЛЕНИЯ]
```

| Аргумент | Описание |
|---|---|
| `-i`, `--info [ПУТЬ]` | Показать информацию об архиве и выйти, ничего не восстанавливая. Без `ПУТИ` — сводка по всем `SOURCE_PATHS` (дата последнего снапшота, размер данных, число файлов/папок). С `ПУТЬЮ` — то же самое для конкретного пути, плюс список файлов/папок верхнего уровня внутри этого снапшота (`kopia ls`, не рекурсивно). |
| `ПУТЬ_ВОССТАНОВЛЕНИЯ` | Необязательный. Каталог, куда восстанавливать данные. По умолчанию — `$RESTORE_ROOT_DEFAULT` (новая папка с меткой времени в `$HOME/Backups/kopia_restore/`). |

### Порядок выполнения

1. Разбор `-h`/`-u`/`-v`; `-i`/`--info` устанавливает режим `MODE=info` и не выходит сразу — обработка откладывается до момента после подключения к репозиторию (см. ниже), чтобы переиспользовать весь код проверки прав/зависимостей/пароля.
2. `read_config_file`.
3. Вывод диагностической информации: путь к репозиторию, конфиг-файл обёртки, конфиг kopia.
4. `require_root "$@"` — как и в `kopia_backup.sh`.
5. `check_required_packages`, `check_kopia_password_file`.
6. Подключение к репозиторию.
7. Если `MODE=info` — вызывается `print_archive_info` (см. ниже) и скрипт завершается; иначе — продолжается обычное восстановление.
8. Определение `RESTORE_ROOT` (аргумент или значение по умолчанию).
9. **Защита от случайной перезаписи**: если целевая папка уже существует и не пуста — запрашивается подтверждение (`y/N`) перед продолжением.
10. Цикл по `SOURCE_PATHS`: для каждого пути ищется ID самого свежего снапшота:
    ```bash
    kopia snapshot list "$SRC" --config-file "$KOPIA_CONFIG" --json \
        | jq -r 'max_by(.startTime) | .id // empty'
    ```
    `max_by(.startTime)` выбирает запись с максимальным временем начала снапшота — результат не зависит от порядка, в котором `kopia` возвращает список.
11. Найденный снапшот восстанавливается в `${RESTORE_ROOT}${SRC}` (структура путей сохраняется, например `/etc` → `$RESTORE_ROOT/etc`).
12. Итог — список путей, которые не удалось восстановить (если такие есть), и код выхода `1`; иначе `exit 0`.

Восстановление всегда идёт в **отдельную** директорию, а не поверх оригинальных путей — это позволяет вручную сверить данные перед тем, как что-либо перезаписывать в рабочей системе.

### Просмотр информации об архиве (`--info`)

```bash
kopia_restore_to.sh --info
kopia_restore_to.sh --info /etc
```

Источники данных, которые использует `print_archive_info()`:
- дата последнего снапшота — поле `startTime` из `kopia snapshot list --json`;
- размер/файлы/папки — поля `stats.totalSize`/`stats.fileCount`/`stats.dirCount`;
- листинг верхнего уровня — `kopia ls <ID>` (не рекурсивно, только непосредственные потомки корня).

Размер выводится в человекочитаемом виде через `numfmt --to=iec-i`, с байтами в качестве резервного варианта, если `numfmt` недоступен.

## 4. Установка (`install.sh`) и автодополнение

Устанавливает `kopia_common.sh`, `kopia_backup.sh`, `kopia_restore_to.sh` и `kopia-completion.bash` в выбранную папку, настраивает права на выполнение и подключает автодополнение bash.

```
install.sh [-h|--help] [--usage] [-v|--version] [ПУТЬ_УСТАНОВКИ]
```

Целевая папка задаётся либо позиционным аргументом (любой произвольный путь), либо интерактивным меню, если аргумент не передан:

```
Выберите папку установки:
  1) /usr/bin/              (требуется root, не рекомендуется)
  2) $HOME/bin/             (рекомендуется, по умолчанию)
  3) $HOME/.local/bin/      (опционально)
  q) Отмена — прервать установку
Ваш выбор (q|1|2|3) [2]:
```

- Пустой ввод (просто `Enter`) → вариант **2** (`$HOME/bin`) по умолчанию.
- `q` — прерывает установку сразу, не создавая ничего.
- Если целевая папка требует root (например, `/usr/bin`) — инсталлятор сам определяет это через реальную проверку прав на запись (не по жёсткому списку префиксов пути) и перезапускается через `sudo`, передавая уже выбранный путь явно, так что меню при этом повторно не показывается.

При переустановке в непустую папку с уже существующими файлами проекта запрашивается подтверждение в том же стиле:
```
Перезаписать существующие файлы? (q|y|n) [N]:
```
Пустой ввод → `N` (не перезаписывать, отмена), `q` — прервать установку с отдельным сообщением.

### Автодополнение (`kopia-completion.bash`)

Дополняет только имена флагов (`--help`, `--usage`, `--version`, `-i`/`--info` и т.д.) — без подстановки конкретных путей. Устанавливается без флага исполнения (подключается только через `source`, как и `kopia_common.sh`).

`install.sh` автоматически:
1. копирует файл в целевую папку вместе с остальными;
2. добавляет строку `source ".../kopia-completion.bash"` в `~/.bashrc` **реального** пользователя (даже если установка шла через `sudo` в `/usr/bin`);
3. перед добавлением проверяет, нет ли такой строки уже — повторные запуски инсталлятора не дублируют строку в `.bashrc`.

После установки нужно перелогиниться либо выполнить `source ~/.bashrc`, чтобы автодополнение заработало в текущей сессии.

## 5. Первоначальная настройка

```bash
# 1. Установить зависимости (или довериться автоматическому предложению скрипта)
sudo apt install jq
# kopia ставится по официальной инструкции: https://kopia.io/docs/installation/

# 2. Создать файл с паролем репозитория
mkdir -p ~/.config/kopia_backup
echo 'ваш-надёжный-пароль' > ~/.config/kopia_backup/.kopia_pass
chmod 600 ~/.config/kopia_backup/.kopia_pass

# 3. Первый запуск — создаст шаблон конфига и остановится
./kopia_backup.sh
# → отредактируйте ~/.config/kopia_backup/kopia_common.conf при необходимости
#   (пути SOURCE_PATHS, REPO_PATH и т.д.)

# 4. Повторный запуск — создаст репозиторий (если его ещё нет) и снапшоты
./kopia_backup.sh
```

### Автоматизация через cron

```
30 2 * * *   /path/to/kopia_backup.sh >>/var/log/kopia_backup.log 2>&1
0  4 * * 6   kopia maintenance run --config-file /home/USER/.config/kopia_backup/repository.config --full >>/var/log/kopia_backup.log 2>&1
```

Если крон-задание должно работать без пароля sudo, добавьте узкое правило в `/etc/sudoers.d/`:
```
username ALL=(root) NOPASSWD: /path/to/kopia_backup.sh, /path/to/kopia_restore_to.sh
```
и убедитесь, что сами файлы `.sh` принадлежат `root:root`, а не пользователю — иначе пользователь сможет подменить содержимое скрипта, который выполняется через `NOPASSWD`.

## 6. Ручное восстановление при аварии (без наших скриптов)

Если система полностью потеряна и есть только физически сохранившееся хранилище репозитория (`REPO_PATH`) и пароль от него — вот последовательность голых команд `kopia`.

### Предпосылки

Нужны обе вещи:
1. Каталог хранилища репозитория (или его копия на другом диске/сервере/облаке).
2. Пароль репозитория — без него восстановление невозможно в принципе, это шифрование, а не просто права доступа.

### Шаг 1. Установить kopia на новой машине

```bash
curl -s https://kopia.io/signing-key | sudo gpg --dearmor -o /usr/share/keyrings/kopia-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/kopia-keyring.gpg] http://packages.kopia.io/apt/ stable main" \
    | sudo tee /etc/apt/sources.list.d/kopia.list
sudo apt update
sudo apt install kopia
```

### Шаг 2. Подключиться к существующему репозиторию (НЕ `create`!)

```bash
kopia repository connect filesystem --path /путь/к/kopia_backup
```
Введите пароль репозитория, когда его запросят.

Если нужно сразу зафиксировать конфиг в конкретном месте:
```bash
kopia repository connect filesystem \
    --path /путь/к/kopia_backup \
    --config-file /root/.config/kopia_backup/repository.config
```

### Шаг 3. Проверить подключение

```bash
kopia repository status
```
Должно показать `Connected to repository`.

### Шаг 4. Посмотреть список источников (путей), которые есть в репозитории

```bash
kopia snapshot list
```

### Шаг 5. Найти нужный снапшот конкретного пути

```bash
kopia snapshot list /etc
```
Возьмите ID нужного (обычно самого свежего) снапшота из вывода.

### Шаг 6. Восстановить данные

**В отдельную папку (рекомендуется для disaster recovery):**
```bash
mkdir -p /mnt/restore
kopia snapshot restore <ID_снапшота> /mnt/restore/etc
```
Дальше вручную сверьте содержимое и скопируйте нужное на постоянные места (`cp -a`, `rsync -a`).

**Сразу «на место» (если система действительно пустая):**
```bash
kopia snapshot restore <ID_снапшота> /etc
```
⚠️ Права и владельцы файлов восстановятся такими же, какими были на момент снапшота.

### Шаг 7. Массовое восстановление всех путей циклом

```bash
for SRC in /etc /var/www /var/spool/cron /home /root; do
    ID=$(kopia snapshot list "$SRC" --json | jq -r 'max_by(.startTime) | .id // empty')
    [[ -n "$ID" ]] && kopia snapshot restore "$ID" "/mnt/restore${SRC}"
done
```

### Альтернатива: смонтировать снапшот без полного восстановления

Если нужно просто посмотреть содержимое или выборочно скопировать пару файлов:
```bash
mkdir -p /mnt/kopia_view
kopia mount <ID_снапшота> /mnt/kopia_view
# смотрите/копируете нужное
umount /mnt/kopia_view
```

### Если хранилище было не локальным (S3/B2/GCS и т.п.)

Меняется только шаг 2, например для S3:
```bash
kopia repository connect s3 \
    --bucket=my-backup-bucket \
    --access-key=... \
    --secret-access-key=... \
    --endpoint=...
```
Остальные шаги (3–7) идентичны — работа со снапшотами не зависит от типа backend-хранилища.

## 7. Известные особенности

- **`kopia snapshot list <путь>` может вернуть снапшоты родительских каталогов**, если они тоже бэкапятся отдельно и вложены друг в друга (см. [kopia#2659](https://github.com/kopia/kopia/issues/2659)). В текущем `SOURCE_PATHS` вложенных путей нет, но при добавлении новых стоит проверять, не оказался ли один путь внутри другого.
- Конфиг-файл (`CONFIG_FILE`) теперь **единый** для обоих скриптов (`kopia_common.conf`) — изменения `SOURCE_PATHS`/`REPO_PATH` достаточно внести один раз, синхронизировать ничего не нужно.
- Все вызовы `kopia` в скриптах используют явный `--config-file`, чтобы не зависеть от конфига kopia по умолчанию (`~/.config/kopia/repository.config`), который может существовать и указывать на другой репозиторий.
